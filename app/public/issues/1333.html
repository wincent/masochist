Bug #1333: Duplicate entries in Atom feeds

Kind bug
Product wincent.com
When 2009-06-26T04:48:29Z
Status closed
Reporter Greg Hurrell
Tags no tags

Description

Sho Fukamachi reports seeing duplicate entries in Atom feeds for the site, and has done some detective work too:

I waited until the phenomenon recurred, then grabbed another copy of the RSS feed. I chose an article which appeared in duplicate and diffed its entries in the old and new files.

I needn't have bothered; the problem is plain as day and would have revealed itself if i'd looked properly:

<     <id>tag:rails.wincent.com,2008:posts/152</id>
---
>     <id>tag:wincent.com,2008:posts/152</id>
5c5
<     <link href="https://rails.wincent.com/blog/why-merging-hurts-in-subversion-but-is-easy-in-git" rel="alternate" type="text/html"/>
---
>     <link href="https://wincent.com/blog/why-merging-hurts-in-subversion-but-is-easy-in-git" rel="alternate" type="text/html"/>
16c16
<     <content src="https://rails.wincent.com/blog/why-merging-hurts-in-subversion-but-is-easy-in-git"/>
---
>     <content src="https://wincent.com/blog/why-merging-hurts-in-subversion-but-is-easy-in-git"/>

And this is my reply:

So if someone accesses the site via rails.wincent.com and is the first to view the feed since the last time the cache was cleared, "rails.wincent.com" will be used in all the entries.

And if someone accesses the site via wincent.com and is the first to view the feed since the last time the cache was cleared, "wincent.com" will be used.

So that explains everything. One of the unexpected consequences of providing two different URLs to access the same application. Doh.

Bugger.

I think I'm going to apply some nginx redirection magic to force all accesses to rails.wincent.com to go to wincent.com. Unless you have a better suggestion.

Thanks for your astute trouble-shooting.
